Message Extract — 主流程文档

用户上传文件，AI 提取结构化信息，合并为统一表格后导出。以下描述完整的 5 步数据处理流程。

流程总览

1 文件上传 POST /api/upload
→
2 模板配置 Schema / Preset
→
3 内容提取 POST /api/extract (SSE)
→
4 对齐合并 POST /api/align-merge (SSE)
→
5 导出结果 XLSX / CSV / JSON

数据在各步骤间的流转

原始文件 → FormData → 服务器临时目录
临时文件 → AI + SSE → {fileId}.json (结构化数据)
各文件结果 → 合并 + 对齐 → mergedExportData (统一表格行)
统一表格 → 客户端生成 → XLSX / CSV / JSON 下载

第一步：文件上传

用户拖拽或选择文件（支持 PDF / 图片 / Word / Excel / 文本）
客户端分块上传（每块 20 个文件，最多 500 个）
FormData POST → /api/upload，逐块等待响应后上传下一块
服务端校验：单文件 ≤ 100MB，总大小 ≤ 2GB，总数 ≤ 500
生成 sessionId + fileId，写入服务器临时目录
返回 { files, sessionId, groupCount } 给客户端
客户端将 file.sessionId 保存到 Zustand，后续提取时传给服务端读取文件
fire-and-forget: cleanupExpiredSessions(cacheExpiryMs)
按用户配置的过期时间（默认 24h）清理过期 session 目录

第二步：模板配置

模板定义输出列（字段名、类型、描述、示例值），指导 AI 按列提取数据。

字段类型说明

模板的作用

有模板 (Schema 模式)

AI 按列名精确提取

键名归一化

AI 返回的键名经常含有垃圾字符（如 "1-基本信息-性别{"），需要在提取和合并阶段清理。三层防线协同工作：

Layer 1: Prompt 约束 system prompt 中禁止 AI 输出单字母(除T/N/M)、纯数字、测量值、日期、{:'" 后缀等 ↓
Layer 2: cleanResultKeys extract 路由写入磁盘前，对每个 key 调用 normalizeKey + isValidKey 过滤 ↓
Layer 3: Merge 阶段过滤 align-merge 路由合并时再次调用 normalizeKey + isValidKey，确保无遗漏

normalizeKey() 处理步骤

输入: "1-基本信息-性别{"
Step 1 trim + NFKC + 零宽字符清除
Step 2 NUMBERED_SECTION_RE: /^(检验结果|检验)\d+\s*[-—]*/
不匹配则跳过
Step 3 数字前缀: /^\d+\s*[-—]\s*/
→ "基本信息-性别{"
Step 4 SECTION_PREFIXES 剥离（最长匹配优先，循环直到无前缀可剥）
匹配 "基本信息-" → → "性别{"
Step 5 cleanKeySuffix: /[{:\'",}\s]+$/
→ "性别"
Step 6 空字符串防护 — 若结果为空返回原始 key
输出: "性别"

isValidKey() 拦截规则

拦截类型被拦截示例原因
空字符串""清理后变空
单个特殊字符",", ")", "→"无意义标点
单字母 (T/N/M 除外)"B", "K"可能是噪声；T/N/M 为 TNM 分期保留
纯整数"1", "1780698"文件 ID 或索引
日期"2026-03-19"不是字段名
时间"7:22", "18:54:06"不是字段名
测量值"4.20mmol/L", "14.1%"是数据值不是字段名
IP 地址"16.22.184.45"不是字段名

错误处理与容错

单文件隔离

提取和合并阶段均使用 workerPool，每个文件/组的处理在独立 try-catch 中执行：

单文件失败 (Extract)

最多重试 3 次 + Schema fallback
最终失败
writeResult(success: false, error) + SSE: file_complete(success: false)
继续处理下一个文件，不影响其他文件

单组失败 (Align-Merge)

mergeGroupWithAI / alignToTemplateWithAI 出错
SSE: group_error
单组失败不影响其他组
客户端可单行重试 (retryGroupIds)

SSE 流断尾防护

流正常结束时追加 '\\n\\n'
强制解析 buffer 中最后一个可能未完成的事件
fallback: 如果 pipelineRowsRef 为空（all_done 丢失）
从 buffer 原始数据中提取；否则使用空结果继续

会话中断恢复

提取过程中每完成一个批次，保存进度到 IndexedDB
页面重新加载时检查是否有未完成的会话
弹出恢复横幅，用户可选择恢复或放弃
过期检查: 超过 24h 的会话拒绝恢复
服务端临时文件已清理，恢复无意义
恢复后从断点继续提取（跳过已有结果的文件）
